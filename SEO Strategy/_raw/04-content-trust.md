## Findings (numbered; each: finding, evidence, SEO impact, recommended fix, priority P0–P3)

1. **Every conversion path funnels to a placeholder WhatsApp number; contact form is a mailto to a placeholder inbox.**
   Evidence: business.ts:12-16 (`'6281234567890' // TODO: real concierge WhatsApp number`, `hello@inkedup.id // TODO: real inbox`); rendered in Footer (Footer.tsx:115,123), Contact page (Contact.tsx:113-134), and used by every wa.me CTA sitewide (waLink, business.ts:30-33). Hardcoded instances bypass business.ts: Home.tsx:735 `phoneNumber="+6281234567890"` (note leading `+` — WhatsApp's wa.me spec requires digits only, so this variant may fail), Artists.tsx:494 `https://wa.me/6281234567890`. Live deployed bundle contains the placeholder (2 matches). Contact form (Contact.tsx:56-64) only opens the visitor's mail client (`mailto:`) — no backend, comment in code admits "No email backend is configured". Studio profiles expose studio numbers that are sequential placeholders (+6281234567891…898, live API).
   SEO impact: critical booking/conversion failure; also feeds the LocalBusiness JSON-LD telephone (+62-812-3456-7890, index.html) — a wrong/fake number in structured data is a trust and local-SEO liability.
   Fix: replace with the real concierge WhatsApp and inbox in business.ts (propagates everywhere); remove hardcoded numbers; give the contact form a real endpoint; update JSON-LD. **P0**

2. **Displayed artist ratings/review counts are unsupported and internally inconsistent with the review data.**
   Evidence: live /api/artists/active shows 4.7–4.9 ratings with 65–154 reviews per artist; but /api/reviews/artist/2 returns exactly one review — anonymous (customer_id null), unattached to any booking (booking_id null), 5/5 in every category, created at seed time (2026-07-11 07:49:50). ArtistProfile.tsx:540-541 renders "Based on {reviewCount} client reviews" while the same section shows zero written reviews ("Detailed written reviews from completed sessions are being collected here", line 543-545). ArtistProfile.tsx:547 claims "Only verified bookings can leave a review" — contradicted by booking_id null. ArtistCard.tsx:60-67 shows rating + count on every card.
   SEO impact: invented-looking review metrics are a textbook trust-claim violation (misrepresentation risk; also disqualifies any future Review/AggregateRating schema). Directly undermines E-E-A-T.
   Fix: show only real, attributable reviews; if none exist, drop rating/review-count UI until real bookings generate them. **P1**

Conversely, the honest live-stat trust bars (Home.tsx:92-97, HowItWorks.tsx:225-244) therefore render "**0+** Sessions Completed" publicly today.
   SEO impact: fabricated volume claims are exactly the "unsupported trust claims" category; the honest counters conversely advertise zero track record. Both damage credibility.
   Fix: delete hardcoded counts; reuse the live-stats pattern everywhere; suppress counters when the value is 0. **P1**
